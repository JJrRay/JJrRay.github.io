MAY 2025
8 MIN READ From pixels to predictions: building a complete FPGA CNN pipeline Architecture · fixed-point design · HLS optimization +

A system-level look at the compute datapath, frame-memory architecture, and software control plane behind a live FPGA inference pipeline.

System architecture and data movement

The accelerator sits inside a complete video system rather than behind a synthetic testbench. HDMI frames enter through the capture pipeline, AXI VDMA writes them into DDR3, and independently managed buffers isolate producer and consumer timing. The inference path selects a stable frame, reduces the region of interest to the network input geometry, and hands the prediction back to the display/control path without stalling video capture.

  • Xilinx Artix-7 on Nexys Video
  • HDMI input and output with AXI VDMA
  • Independent capture, inference, and display buffers

Compute mapping and numerical format

The CNN was converted from a software-oriented layer graph into a static fixed-point datapath. Convolutional reuse, pooling, and dense-layer accumulation were expressed in synthesizable C++ and shaped with loop pipelining, selective unrolling, local buffering, and array partitioning. Those transformations expose MAC-level parallelism while keeping memory ports and DSP pressure within the Artix-7 resource envelope.

  • Static fixed-point activations, weights, and accumulators
  • Pipelined loops with selective unrolling and partitioning
  • 0.318 µs measured inference latency

Control plane and synchronization

MicroBlaze firmware owns accelerator sequencing and buffer routing: it programs frame addresses, selects the inference source, launches the HLS IP, polls completion, and publishes the classification result. AXI-Lite carries low-bandwidth control traffic while the DDR/VDMA path handles bulk frame movement. Triple buffering prevents capture from overwriting a frame that inference or display still owns.

  • MicroBlaze C control firmware
  • AXI-Lite register control and memory-mapped frame access
  • Explicit ownership across asynchronous pipeline stages

Performance interpretation

The reported 0.318 µs figure is accelerator-kernel latency, not end-to-end camera latency; the latter also includes frame capture, memory traversal, preprocessing, and display synchronization. Against the project’s Ryzen 7 5800X3D reference implementation, the kernel delivered the reported 4,000× latency improvement and 95% lower power. The useful engineering result is the integration: compute, memory traffic, and software control were closed as one deployable system.

  • Kernel latency distinguished from system latency
  • Resource-aware parallelism on a mid-range FPGA
  • Measured hardware/software baseline comparison

DEC 2025
11 MIN READ Speculate, select, commit: decoding market data without waiting Low latency · protocol design · coverage +

A look inside a parallel decoder that trades a small amount of area for deterministic latency—and the verification strategy used to trust it.

Speculate before the message is known

Instead of identifying an ITCH type and then dispatching it, all nine decoders inspect the stream in parallel. A matching decoder continues capturing fields while every non-matching decoder enters suppression for the known message length. Canonical arbitration accepts the result only when exactly one internal-valid signal is asserted.

  • Nine ITCH 5.0 message types
  • One-hot canonical output arbitration
  • No idle cycle required between messages

My extension work

The base architecture was extended with Add Order MPID, Executed Order with Price, and Broken Trade decoders. Six existing modules were refactored from multiple signal-driving processes into single synthesizable processes for Vivado compatibility. The integration was then wrapped with AXI4-Stream for input data and AXI4-Lite for result and control access.

  • Three new message decoders
  • Single-process Vivado synthesis refactor
  • AXI DMA and register interface on PYNQ-Z2

Verification in four layers

Correctness was checked against a Python golden model, then exercised in Cocotb with random field values, permuted sequences, and back-to-back traffic. AXI behavior was verified again in Vivado simulation before the design was validated on PYNQ-Z2 using real DMA transfers. Mid-packet valid drops explicitly abort active decoders so recovery does not corrupt the next message.

  • Golden model and field-level scoreboard
  • Cocotb assertions and randomized payloads
  • On-target PYNQ hardware validation

Measured implementation

At 100 MHz, complete-message latency scales with packet length: Delete Order completes in 100 ns while the 40-byte Trade and Add Order MPID messages complete in 410 ns. The implementation uses no BRAM or DSP resources because parsing is dominated by registers, control, and field assembly.

  • 2,578 LUTs (4.85%)
  • 3,705 flip-flops (3.48%)
  • 0 BRAM and 0 DSP blocks

DEC 2024
7 MIN READ What changed when my RISC-V core met physical design RTL-to-GDS · STA · design for test +

Moving beyond clean RTL: resolving hazards, reading timing reports, inserting scan chains, and closing the loop between logic and layout.

IFFetch→IDDecode→EXExecute→MEMMemory→WBWrite back

A complete five-stage machine

The core implements the classic Fetch, Decode, Execute, Memory, and Write Back pipeline in modular VHDL. It includes a 32×32-bit register file with x0 hardwired to zero, immediate generation, branch and jump target calculation, ALU operations, and separate instruction and data-memory interfaces.

  • RV32I 32-bit datapath
  • Five pipeline stages
  • 32 architectural registers

Instruction-level validation

Assembly programs exercise arithmetic, logical, shift, comparison, branch, jump, load/store, and upper-immediate operations. The simulation environment includes basic instruction tests and a Fibonacci program that checks the processor through F(20)=6765 rather than validating modules only in isolation.

  • ADD/SUB and immediate operations
  • JAL/JALR plus conditional branches
  • LW/SW memory behavior

When RTL meets the physical world

The design was synthesized with Cadence Genus and placed and routed with Innovus using the GPDK045 educational process. Static timing analysis was performed at the 0.9 V, 125°C worst-case corner, forcing the pipeline and constraints to be evaluated as a physical implementation rather than an ideal behavioral model.

  • 45 nm standard-cell flow
  • 1,861 synthesized cells
  • 7,815.14 µm² total core area

Design for test

A scan chain was inserted so internal sequential state could be controlled and observed during manufacturing test. Functional behavior was rechecked with assembly-driven and timing-constrained simulations after synthesis and implementation, closing the loop across RTL, netlist, timing, and testability.

  • Scan-chain DFT
  • Post-synthesis functional verification
  • Timing-constrained simulation
